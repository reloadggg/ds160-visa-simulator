# DS-160 模拟器使用体验评估

日期：2026-04-18  
评估视角：首次使用的普通用户  
评估对象：当前 `.worktrees/ds160-simulator-v1/` 版本

## 1. 结论

当前版本从产品使用体验上看，`理论上能跑完`，但 `普通用户不太可能自然顺畅跑完`。

原因不是主流程不存在，而是主流程对环境和操作习惯有较强依赖：

- 用户必须先选择签证家族。
- 用户必须按系统要求上传必需材料。
- 上传后的解析任务必须被后台 worker 实际消费。
- 文件名和材料类型需要刚好符合当前系统的识别方式。

如果上述条件都满足，流程可以走到正式 interview 和报告查看阶段。  
但只要其中任何一项不满足，用户就很容易停留在 `gate_review` 阶段，反复收到“继续补材料”或“等待解析”的提示。

## 2. 当前主流程

当前系统面向用户的主流程可以概括为：

1. 进入聊天界面。
2. 选择签证家族。
3. 系统创建 session，并给出必需材料包。
4. 用户上传材料。
5. 系统等待材料解析完成。
6. gate 通过后进入正式 interview。
7. 用户继续问答。
8. 用户查看用户报告或内部报告。

从接口和测试设计上看，这条链路是成立的。

## 3. 能跑完的前提条件

当前版本要“顺利跑完”，至少需要满足以下前提：

- 后台存在实际执行的解析 worker，而不只是把任务写入队列。
- 用户上传的材料类型与当前 session 所需材料一致。
- 用户上传的文件名最好与内部 `document_type` 命名接近。
- 用户能理解 gate 阶段不是正式问答阶段，愿意先补材料再继续。

这意味着系统目前更适合内部测试、演示或由熟悉流程的人操作，不太适合完全陌生的新用户直接上手。

## 4. 主要体验问题

## 4.1 上传后可能一直卡住

这是当前最严重的问题。

上传文件后，系统会创建 `gate_parse` 任务，但当前主应用里没有看到自动启动后台解析消费器的逻辑。  
如果没有额外 worker 进程主动消费这些任务，用户会一直停留在“等待解析”状态。

对用户来说，感知会是：

- 我已经上传了文件。
- 系统也提示“材料已接收”。
- 但我再发消息，系统只会重复说“正在解析”。

这会让用户误以为系统坏了，或者不知道自己还需要做什么。

## 4.2 材料识别依赖文件名，容易误判

当前 gate 检查材料是否匹配时，核心匹配逻辑仍然依赖文件名中是否包含 `document_type`。

这会带来一个明显问题：

- 用户在 UI 中明明已经选择了材料类型。
- 但如果文件名不是系统预期的格式，例如 `bank-statement-final.pdf`，系统可能仍然不把它识别为 `funding_proof`。

从用户角度看，这属于“我明明按要求做了，但系统不认”，体验非常差。

## 4.3 gate 过硬，用户会觉得系统不会聊天

当前 gate 没通过前，用户发送任何消息，系统都会优先要求补材料，而不会进入正式问答。

这在规则设计上是合理的，但在体验上会造成两种认知落差：

- 用户已经开始描述自己的情况，但系统完全不接话。
- 用户以为这是一个“聊天式面签模拟器”，但系统在前期更像“材料检查机器人”。

如果没有明确的阶段说明，第一次使用的人会觉得产品与预期不一致。

## 4.4 等待解析时缺少进度感

当前用户在等待解析时，主要只能看到“documents are waiting to be parsed”这类提示。

缺少以下关键信息：

- 已上传了几份材料。
- 还缺哪几份材料。
- 哪些材料已经识别成功。
- 当前是在排队、解析中，还是解析失败。

没有进度感时，用户很容易连续重复上传、重复发送消息，进一步放大挫败感。

## 4.5 报告能看，但不够产品化

当前 Chainlit 中“查看用户报告”和“查看内部报告”本质上是直接展示后端返回的字典对象。

这说明：

- 功能存在。
- 但展示形式仍然更偏工程调试，而不是面向普通用户的阅读体验。

尤其内部报告，更像研发追踪对象，不适合直接暴露给非技术角色。

## 4.6 OpenAI-compatible 更像技术适配，不像真实连续会话

`/v1/chat/completions` 当前是每次请求都新建一个 session，只取最后一条 user message。

这意味着如果外部前端把它当作标准连续对话接口使用，实际体验会和用户预期不一致：

- 用户以为是在连续对话。
- 后端实际是在重复创建新会话。

这个问题对内部开发测试影响不大，但对未来对外接入会形成明显产品风险。

## 5. 整体判断

如果从“用户能否理论完成流程”看，答案是 `能`。  
如果从“陌生用户第一次进入是否大概率完成”看，我的判断是 `不能保证`。

更准确的说法是：

- 这是一个已经具备完整业务骨架的 Alpha 原型。
- 它适合由熟悉系统的人进行演示、测试和产品验证。
- 它暂时还不具备“自然用户可无障碍完成”的体验成熟度。

## 6. 优先级建议

如果从使用体验出发，建议按以下顺序优化：

1. `P0`：补上自动消费解析任务的正式运行机制，避免用户永远卡在 waiting for parse。
2. `P0`：gate 材料匹配改为优先使用上传时的 `document_type`，不要继续依赖文件名。
3. `P1`：在 gate 阶段增加明确进度反馈，例如“已收到 3/5 份材料，2 份待解析”。
4. `P1`：增强阶段提示，让用户知道当前是“材料门控阶段”还是“正式 interview 阶段”。
5. `P1`：把用户报告改造成可读摘要，而不是直接展示原始字典。
6. `P2`：重新定义 OpenAI-compatible 会话语义，避免未来接入方误用。

## 7. 最终评价

当前版本不是“流程跑不通”，而是“流程对正确操作和正确环境依赖太强”。  
真正的问题不在业务主链路缺失，而在于：

- 缺少自动推进机制。
- 缺少可理解的状态反馈。
- 缺少对真实用户操作习惯的容错。

从产品视角看，下一阶段最值得投入的不是继续扩更多签证家族，而是先把 `一个家族的一整条使用链路` 做到稳定、可理解、可完成。
