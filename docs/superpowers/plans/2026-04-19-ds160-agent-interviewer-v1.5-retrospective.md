## 二、执行摘要

这次 v1.5 的核心，不是继续修一个“会聊天的打分系统”，而是把产品主线重新拉回“更像真人面签官”的方向。

项目最终完成了四个关键转向：

1. 从“单轮处理”转向“本次会话连续记忆”
2. 从“材料门控优先”转向“对话主线 + 材料作为证据”
3. 从“打分器主导”转向“面签官 owner 主判断”
4. 从“硬编码提示词”转向“可配置 prompt 体系”

最终系统已经具备以下主线能力：

1. 只记住本次会话，不引入跨会话人物记忆
2. 能把对话与材料放在一起理解
3. 能在材料未齐时继续进入实质问答
4. 能要求当前唯一关键证明，而不是同时撒网补件
5. 能把高风险复核和模拟拒签明确分层
6. 能把 scoring 收回后台辅助层，不再对用户输出分数口吻

## 三、为什么必须重做主线

项目启动时，现有版本暴露出四类根问题：

1. 会话是“伪连续”的，很多入口每次都像重新开始
2. 系统体验更像材料审核流程，不像面签官
3. gate、scoring、规则链路过强，Agent 主判断过弱
4. prompt 与行为大量写死在代码里，产品迭代成本高

从产品视角看，用户应该感受到的是“同一个签证官一直在听、在看、在追问、在判断”，而不是“每一轮被不同小模块机械接力”。

因此这次重构不是局部打补丁，而是把系统重心重新定义为：

`本次会话记忆 -> 材料理解 -> 面签官主判断 -> 用户可理解结果`

## 四、目标与边界

### 4.1 本轮目标

1. 建立本次会话记忆底座
2. 让材料未齐时也能继续进入实质问答
3. 建立唯一的 interviewer owner
4. 明确状态模型、高风险与拒签分层
5. 补强材料理解、材料反馈与关键证据链路
6. 完成 prompt 配置化
7. 把 scoring 收回后台辅助层
8. 让需求、实现、测试三者对齐

### 4.2 明确不做

1. 跨会话人物记忆
2. 跨会话材料复用主逻辑
3. 真实案例 RAG 主链路
4. 向量库作为主底座

这几条边界在整个执行周期内保持不变，也是避免产品再次“做歪”的核心约束。

## 六、关键架构变化

### 6.1 会话从“最后一句”升级为“本次会话记忆”

Task 1 建立了 `session_turns` 表和 interviewer 状态字段，让系统能持续保存：

1. 用户说过什么
2. 系统问过什么
3. 当前关键焦点是什么
4. 当前 interview 状态是什么

这一点直接纠正了“每轮像新会话”的核心体验问题。

### 6.2 gate 从硬拦截器降为支持层

Task 2 把 gate 从“是否允许进入面谈”的主流程开关，改成了“当前材料 readiness 的支持信息”。

结果是：

1. 材料未齐时，用户仍可进入实质问答
2. 系统仍能提示最缺什么关键证明
3. `gate_progress` 继续存在，但不再等于“不能开始”

这一步是把产品从“流程系统”拉回“面签系统”的关键拐点。

### 6.3 interviewer owner 成为唯一主判断层

Task 4 后，系统明确只有 interviewer owner 负责：

1. 决定当前最关键问题
2. 决定下一步是继续问、补件、高风险还是拒签
3. 写入 `current_focus`

原来的 extractor、question、scoring 不再是“并列主脑”，而是 interviewer owner 的辅助组件。

### 6.4 状态模型从工程态转成产品态

Task 5 把外部状态整理成用户和产品都能理解的状态集合，例如：

1. `continue_interview`
2. `verify_key_issue`
3. `waiting_key_proof`
4. `high_risk_review`
5. `simulated_refusal`

这一步让报告、API、前端体验终于能围绕统一状态表达，而不是散落在内部字段里。

### 6.5 prompt 完成配置化

Task 12 把 prompt 从硬编码迁到 YAML，并支持：

1. base prompt
2. family override
3. runtime 注入 instructions

这意味着后续产品想调整签证官口吻、判断规则或家族差异，不再需要直接改 agent 代码。

### 6.6 scoring 被收回后台辅助层

Task 13 是本轮最后一个真正的高风险架构收口点。

最终收口后的原则是：

1. scoring 只保留内部辅助价值
2. scoring 不再决定对用户说什么
3. scoring 不再单独点燃高风险或拒签
4. scoring 不再拥有 `current_focus` 控制权
5. 对外 `score_summary` 清空，不再把分数包装成主文案

## 七、最难收口的两个阶段

### 7.1 Task 11：上传反馈与主线保持

Task 11 的难点不是“识别文件”，而是“识别后不要把主线带偏”。

这轮收口后的行为是：

1. 明显无关材料返回 `not_helpful`
2. 有帮助但不是当前主线的材料返回 `partial_helpful`
3. 有帮助且命中当前关键证明的材料返回 `helpful`
4. `counts_toward_gate=False` 的材料不会错误地推动主线

这里最大的价值，是让“材料上传”从单纯的文件动作，变成了主线判断的一部分。

### 7.2 Task 13：把打分退回辅助层

Task 13 是全程反复审查次数最多的任务之一，主要因为“表面降级 scoring”和“真正拿掉 scoring 主导权”是两件不同的事。

本轮最终修掉了三类关键问题：

1. `need_more_evidence` 下具体补件目标不能丢失
2. `HIGH_RISK_REVIEW` 不能被任意高危 score blanket override
3. redline refusal 不能由 score-only 信号单独触发

最终形成的判定链路是：

1. refusal 必须经过 owner findings 形成 candidate
2. governor 再决定是否进入 refusal
3. score 只能作为内部风险信号与确认信息，不能越权替代 owner judgement

## 八、审查中暴露出的典型问题

### 8.1 容易出现的伪收口

这次最有代表性的经验是：很多问题“看起来已经修了”，但实际上只是把症状从一个地方搬到另一个地方。

典型例子包括：

1. 表面把 `score_summary` 清空了，但真实 API mock 仍可能回传旧值
2. 表面限制了 high risk review，但 refusal 仍能从 score-only 风险标记点燃
3. 表面有 requested document，但在 fallback 分支里仍可能丢失具体 document

这说明收口不能只看局部 diff，必须看真实主流程和兜底路径。

### 8.2 回归问题往往来自测试隔离而不是业务代码

Task 14 最后阶段，全量 `not live_llm` 回归里出现过 parse worker 用例串扰。

根因不是主线逻辑变了，而是测试运行过程中遗留 runtime worker 状态导致手动 worker 用例被提前消费。

最终通过显式停止遗留 parse worker runtime，才让这类整套回归中的非确定性失败被收住。

这次的教训是：

`任务收口不只要修业务，还要修测试环境与状态隔离。`

## 十一、还存在哪些非阻断风险

当前还有两类值得记住的非阻断风险：

1. 公共 API 边界层本身还没有强制剥离 `score_summary`
   - 目前真实主流程不泄露，主要依赖 interviewer runtime 主动返回空对象
   - 后续如果有人在边界层或 mock 里恢复旧行为，可能出现回漏
2. live LLM 测试尚未在具备凭据的环境里补跑
   - 当前只能确认非 live 环境下需求、实现、测试已经对齐
   - 还不能把“真实模型联调通过”写成结论

## 十二、下一阶段建议

如果继续推进下一阶段，我建议按下面顺序走：

1. 在具备凭据的环境补跑 live LLM 测试
2. 给公共 API 边界层增加更硬的 `score_summary` 防回漏兜底
3. 继续观察 prompt 配置化后的迭代效率与行为稳定性
4. 评估是否进入下一阶段的真实案例与检索层建设

这里要强调：

下一阶段可以扩展，但不要反向破坏本轮已经建立的主线边界。
