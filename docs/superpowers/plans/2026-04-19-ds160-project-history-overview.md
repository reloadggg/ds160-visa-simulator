## 二、先说结论

如果只用几句话概括这个项目从创立到现在的变化，可以这样理解：

1. 项目最初是一个后端优先的 DS-160 面签模拟器
2. 初版很快把 API、上传、评分、报告、OpenAI-compatible 适配层都搭了起来
3. 但在真实体验上，系统更像“材料流程 + 打分系统”，不像“同一个面签官在持续判断”
4. 因此在 v1.5 阶段，项目进行了方向性重构，把主线改成“本次会话记忆 + 材料证据 + 面签官主判断”
5. 截至现在，主线开发已经完成，非 live 验收通过，项目处于“可以作为当前主线版本交付，但还需要补 live 环境验证”的状态

## 三、项目为什么会创立

项目创立的直接目标，是构建一个围绕 DS-160 场景的美国非移民签证面签模拟器。

从最初设计看，这个项目要解决的不是一个普通聊天机器人问题，而是一个更具体的签证模拟问题：

1. 用户需要提交对话信息
2. 用户需要上传材料
3. 系统要把材料、事实、证据、风险和最终结论区分开
4. 系统既要像问答系统，也要像一个有边界的面签判断系统

在最初阶段，项目采用的是比较典型的“后端优先 + 模块化单体”策略，先把后端 API、数据结构、材料处理、打分与 governor 这些基础骨架搭起来。

## 四、项目创立阶段：v1 的形成

### 4.1 起点

从仓库历史看，最早的记录是 2026-04-17，首先落下的是：

1. v1 设计文档
2. v1 实现计划

这说明项目一开始就不是无计划编码，而是先定规格，再分任务推进。

### 4.2 初版建设重点

v1 阶段的核心建设内容，主要集中在以下几类：

1. FastAPI 单体服务骨架
2. 核心 domain contracts
3. persistence models 和 repositories
4. policy pack 与 runtime policy registry
5. 文件上传、解析任务、异步 parse job
6. session / gate API
7. scoring 与 governor service
8. message orchestration
9. reports 与 OpenAI-compatible adapter
10. live LLM integration 测试基础设施

换句话说，v1 的主要目标是：

`先把一个完整可运行的后端产品骨架搭起来`

### 4.3 初版的价值

初版做对了几件重要的事：

1. 很快形成了完整的服务形态，而不是停留在 demo
2. 事实、证据、提议、最终决策被分层存放
3. 对外接口、上传流程、解析任务、报告输出都具备了基本闭环
4. 为后续更深的行为重构留下了明确的代码落点

也正因为 v1 很快具备了“端到端骨架”，团队才能在很短时间里发现真正的问题不在“有没有功能”，而在“主语义是不是对”。

## 五、项目第二阶段：能力补全与体验暴露

### 5.1 2026-04-18 的扩展建设

在 2026-04-18，项目进入了一个快速补强阶段，主要做了这些事：

1. evidence foundation
2. PydanticAI runtime cutover
3. formal runtime
4. Chainlit UI 接入
5. typed multimodal uploads
6. structured evidence extraction
7. gate flow automation and feedback

这说明项目从“有基础骨架”进一步进入“把材料、运行时、UI、LLM 工具链真正接起来”的阶段。

### 5.2 也是在这个阶段，问题开始暴露

功能补上后，项目没有变得更轻松，反而更清楚地暴露出体验问题。

最关键的一份信号是 2026-04-18 的 UX Flow Evaluation。它指出了几个方向性问题：

1. gate 太硬，用户感觉自己还没开始面谈就被拦住
2. 文件上传和解析的主线体验不够顺
3. 材料匹配与进度反馈不够稳
4. 系统更像流程控制器，不像面签官

这一步非常关键，因为它把团队从“继续补功能”拉回到“要不要重定义产品主线”这个问题上。

## 六、项目第三阶段：v1.5 的方向性重构

### 6.1 为什么必须做 v1.5

到了这个阶段，项目已经不是“缺功能”，而是“主方向有偏差”。

如果继续按原思路往前补，系统会越来越像：

`材料审核流程 + 风险评分系统 + 固定追问器`

而不是：

`一个像真人面签官一样，有连续记忆、能看材料、会追问、会判断的 Agent`

所以 v1.5 的核心，不是继续修小 bug，而是把产品方向重新定义。

### 6.2 v1.5 的重新定义

v1.5 最重要的重定义有三条：

1. 只记住本次会话，不做跨会话人物记忆
2. 拒签依据必须来自本次会话资料和对话
3. 数据库是本次会话容器，判断主体必须是 interviewer owner，而不是隐藏在库里的规则黑盒

系统主线也被明确改写为：

`本次会话记忆 -> 材料理解 -> 面签官主判断 -> 用户可理解结果`

## 七、v1.5 期间到底改了什么

这次重构被拆成了 14 个任务，但如果用更容易理解的方式，可以把它分成五个阶段。

### 7.1 阶段一：把记忆和主线立起来

包括：

1. 建立 `session_turns`
2. 补 `interviewer_state_json`
3. 修 OpenAI-compatible 连续会话语义
4. 拆掉 gate 硬拦截

这部分的结果是：

系统终于开始具备“本次对话一直在延续”的能力。

### 7.2 阶段二：建立 interviewer owner

包括：

1. interviewer runtime service
2. 统一的面签状态模型
3. `current_focus` 写入权回收

这部分的结果是：

系统第一次真正拥有了“唯一主脑”。

### 7.3 阶段三：把材料变成证据，而不是附件

包括：

1. document pipeline 补强
2. profile recompute
3. retrieval / evidence 层补强
4. 材料引用、冲突处理、追问逻辑补强

这部分的结果是：

系统开始真正把上传材料放进判断过程，而不是“传完就算完成动作”。

### 7.4 阶段四：把产品可配置能力补齐

包括：

1. prompt 从代码迁移到 YAML
2. base + family override
3. Agent runtime 指令外部注入

这部分的结果是：

产品后续修改“签证官角色提示词”不再需要直接改 agent 代码。

### 7.5 阶段五：把 scoring 收回后台

包括：

1. scoring 不再决定对用户说什么
2. scoring 不再控制 `current_focus`
3. scoring 不再直接升级高风险或拒签
4. 对外 `score_summary` 清空

这一步是整个 v1.5 最关键的一次主语义纠偏。

## 八、从项目管理视角看，这个项目经历了什么变化

### 8.1 从“快速搭骨架”到“快速暴露问题”

最开始几乎是在极短时间里把完整系统骨架建起来，这个阶段速度很快。

优点是：

1. 很快得到一个完整可运行系统
2. 很快能开始真实测试

代价是：

1. 主线问题不会在设计图里暴露，而会在交互体验里暴露

### 8.2 从“补功能”到“改方向”

很多项目卡在这里，会继续做增量修修补补。

这次项目比较好的地方是，没有在错误方向上继续加码，而是在 UX 评估暴露问题后，及时把主语义重定向到了 interviewer owner 模式。

### 8.3 从“模块并列”到“owner 中心”

早期更像：

1. gate
2. extractor
3. scoring
4. governor

多个组件并列拼接。

现在更像：

1. interviewer owner 负责当前判断
2. 其他模块都退为辅助层

这意味着系统的控制权终于统一了。

`主线已完成，非 live 已验收，live 待环境验证`

## 十、当前版本最重要的特征是什么

如果现在要向别人介绍这个项目，不应该再用“一个带上传和打分的签证模拟器”去描述它。

更准确的说法应该是：

`这是一个围绕 DS-160 场景构建的面签 Agent 系统。它以本次会话为边界，持续记住对话与材料，把材料作为证据，由 interviewer owner 统一判断下一步是继续问、补件、高风险复核还是模拟拒签。`

这里最重要的不是“它能聊天”，而是：

1. 它记得本次会话
2. 它看得见材料证据
3. 它有统一判断 owner
4. 它已经从流程系统转向 Agent 判断系统

## 十一、当前还需要注意什么

虽然主线已完成，但仍有两类值得持续观察的点：

1. live LLM 环境补测
2. API 边界层对 `score_summary` 的长期防回漏

前者决定真实模型联调稳定性，后者决定未来重构时是否会悄悄把 scoring 再暴露回用户层。

## 十二、下一步怎么看

如果后面要继续推进，我建议用这样的视角看项目：

### 12.1 这不是“重做”，而是“在正确主线上继续迭代”

当前主线已经被拉正，后续工作不应该再回头争论“到底要不要做流程 gate”或者“要不要让 scoring 再回到主脑位置”。

### 12.2 下一阶段更适合做环境验证和能力扩展

更合理的下一步包括：

1. 在可用凭据环境补跑 live LLM 测试
2. 强化边界层兜底
3. 评估真实案例与检索层是否进入下一阶段

### 12.3 一定要继续守住当前产品边界

尤其是这三条：

1. 只记住本次会话
2. 拒签必须来自本次会话里的资料和对话
3. interviewer owner 必须继续是唯一主判断层

## 十三、最后的总结

这个项目从创立到现在，只用了很短时间就经历了三个很典型的阶段：

1. 快速搭系统
2. 快速暴露主线问题
3. 快速完成方向性重构

这其实是一个很好的信号。

因为这说明项目不是停留在“堆功能”，而是在很早阶段就抓住了真正的问题：

`一个签证面签模拟系统，最重要的不是功能堆得多快，而是主语义必须像一个真正的面签官。`

截至现在，这个项目已经从最初的 v1 骨架，走到了一个更清晰的 v1.5 形态。

它还没有完成所有未来阶段工作，但至少已经完成了最重要的一件事：

`把系统拉回了正确方向。`
