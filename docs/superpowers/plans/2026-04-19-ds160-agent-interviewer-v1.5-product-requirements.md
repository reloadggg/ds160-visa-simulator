# DS-160 Agent 面签官 v1.5 产品需求稿

日期：2026-04-19

状态：已完成非 live 验收（live 待环境验证）

## 1. 这次 v1.5 到底要做什么

这次不是继续修一个“会聊天的打分系统”。

这次要把产品拉回正确方向：

做一个更像真人面签官的 Agent。

它要做到的不是单次回答问题，而是：

1. 在同一场会话里持续记住用户说过的话
2. 记住用户上传过的材料
3. 把“口头解释”和“书面材料”放在一起看
4. 根据前后信息继续追问
5. 在发现明显问题、红线问题、造假问题时，能给出拒签判断
6. 在材料不足时，不是假装已经懂了，而是明确要求补材料

一句话总结：

`v1.5 要做的是 Agent 面签官，不是流程系统外面套一层聊天壳。`

## 2. 我们希望用户感受到什么

用户进入系统后，应该感受到的是：

1. 我面对的是同一个“面签官”
2. 这个面签官记得我刚才说过什么
3. 这个面签官也记得我上传过什么材料
4. 它问我的下一句话，和前面的内容是连着的
5. 它会参考材料，也会质疑解释
6. 它不是只会提醒“请上传文件”，而是真的在判断我的案子

如果用户感受到的是：

1. 每一轮都像重新开始
2. 我刚刚说的话下一轮就忘了
3. 上传材料只是走流程，不真的影响问答
4. 系统只会机械地补件和打分

那就说明产品仍然做歪了。

## 3. 当前版本为什么不对

用产品语言说，当前版本主要有 4 个问题：

### 3.1 它不像同一个面签官

用户以为自己在持续对话，但系统很多时候并没有真的延续同一场会话。

### 3.2 它更像材料审核流程，不像面签

现在系统很强调“材料齐不齐”，但真人面签不是只看 checklist。

真人面签会：

1. 听你说
2. 看你交的材料
3. 比较前后是否一致
4. 再决定继续问、要求补充、还是直接拒签

### 3.3 它更像打分器，不像判断者

现在“打分、门控、规则”太强，“Agent 主动判断”太弱。

这会让产品越来越像：

`材料系统 + 风险评分 + 固定追问`

而不是：

`一个有记忆、有判断、有追问能力的面签官`

### 3.4 很多规则被写死了，不方便产品迭代

签证判断里有很多东西不是固定死规则。

很多判断要靠：

1. 提示词
2. 经验
3. 真实案例
4. 最新政策理解

如果这些东西改一次就要改代码，那产品迭代会非常慢，也不适合后面接真实案例。

## 4. v1.5 的产品定位

v1.5 的产品定位是：

`一个以对话为主线、以材料为证据、以判断为目标的 DS-160 模拟面签 Agent。`

这里有 3 个关键词。

### 4.1 对话为主线

整场过程必须是一条连续主线。

不是每一轮都单独处理。

### 4.2 材料为证据

材料不是“上传完成即可”。

材料必须能被记住、被引用、被用于判断。

### 4.3 判断为目标

产品目标不是把分数算出来。

产品目标是：

1. 判断是否继续问
2. 判断是否要求补充说明或补材料
3. 判断是否出现重大风险
4. 判断是否应该拒签

## 5. v1.5 必须做到的事情

这是 v1.5 的必做项。

## 5.1 必须有连续上下文

同一场会话中，系统必须一直记住：

1. 用户之前说过什么
2. 用户之前上传过什么
3. 系统之前问过什么
4. 系统之前已经形成了什么判断

不能只记最后一句。

## 5.1.1 已确认：只需要记住本次会话，不做跨会话的人物记忆

这一点非常重要。

v1.5 采用的不是：

`系统长期记住这个用户是谁、以前来过几次、以前被怎么判断过`

而是：

`系统只需要记住本次会话里的内容`

这代表：

1. 不做用户级长期记忆
2. 不做跨会话的人物档案
3. 不做“因为你以前怎样，所以这次直接怎样”的感觉
4. 系统真正需要记住的是本次面谈里发生了什么

所以 v1.5 记住的重点是：

1. 本次会话说过什么
2. 本次会话传过什么材料
3. 本次会话已经出现了什么冲突
4. 本次会话当前走到了哪一步

简化理解就是：

`记住这次，不记住这个人一辈子`

## 5.2 必须同时记住“话”和“材料”

系统的记忆不能只记对话，也不能只记材料。

它必须把两类信息放在一起：

1. 用户解释
2. 用户材料

然后用这两类信息继续判断。

## 5.2.4 已确认：拒签依据必须来自本次会话的资料和对话，而不是隐藏数据库历史

这一点也必须写清楚。

v1.5 采用的不是：

`系统因为数据库里存过一些旧信息，就直接知道该不该拒签`

而是：

`系统的拒签依据必须来自本次会话里的资料和对话`

这代表：

1. 拒签必须能解释到本次会话里看得到的东西
2. 这些依据可以来自：
   - 用户当前会话里的说法
   - 用户当前会话里上传的材料
   - 当前会话里出现的冲突
   - 当前会话里长期不给关键证明
   - 当前会话里持续回避核心问题
3. 不能让用户感觉系统是“从数据库暗中知道了一些别的东西”

简化理解就是：

`数据库可以是存放容器，但不能成为隐藏判断来源`

## 5.2.1 已确认：口头解释和材料冲突时，两边都不先完全相信

这一点也已经确认。

v1.5 采用的不是：

`只要用户说了新解释，就立刻覆盖材料`

也不是：

`只要材料和用户说法不一致，就立刻认定材料一定对`

而是：

`当口头解释和材料冲突时，系统先把它当成冲突点，继续核实`

这代表：

1. 系统不会轻易站队
2. 系统不会因为一句解释就推翻材料
3. 系统也不会机械地把材料当成永远正确
4. 系统会围绕这个冲突继续追问、继续要求证明

这更接近真人面签的处理方式：

1. 先识别冲突
2. 再继续核实
3. 如果能解释清楚并补上证明，可以继续
4. 如果解释不清、证明不足、或冲突非常明显，就进入高风险或拒签判断

## 5.2.2 已确认：允许用户改口，但必须讲得通，才能说服系统

这一点也已经确认。

v1.5 采用的不是：

`用户前面说过的话，一旦说出，就永远不能改`

也不是：

`用户后面只要换一种说法，系统就直接接受新版本`

而是：

`允许用户改口，但新解释必须符合逻辑，才能说服系统`

这代表：

1. 用户可以补充、修正、重新解释
2. 系统不会因为“改口”就立刻判错
3. 但系统也不会把前面的说法直接抹掉
4. 前后变化会成为重点核实对象

这也意味着：

1. 旧说法要被记住
2. 新说法要和上下文讲得通
3. 必要时还要补证明
4. 如果改口越来越不合理，就会变成风险点

简化理解就是：

`可以改口，但要讲得通，还要经得住核实`

## 5.2.3 已确认：系统追问时要明确引用材料

这一点也已经确认。

v1.5 采用的不是：

`系统只泛泛地提问，不点出它看到的材料内容`

而是：

`系统在追问时，要明确引用材料`

这代表：

1. 用户会明确知道系统真的看过材料
2. 系统会把材料和口头解释放在一起比较
3. 追问会更具体、更像真人面签

例如，系统应该能问：

1. 你上传的材料里写的是 A，但你刚才说的是 B，请解释
2. 你提交的证明里没有看到你刚才提到的内容，请补充说明
3. 你这份材料支持了部分说法，但关键点还没有被证明

简化理解就是：

`不是空问，而是带着材料来问`

## 5.3 必须会追问

系统不能只会：

1. 给分
2. 提醒补件
3. 输出固定文案

它必须会像面签官一样继续追问。

比如：

1. 你说父母资助，那具体谁出钱
2. 你说去读书，那为什么选这个学校
3. 你材料里写的是 A，你现在说的是 B，为什么不一致

## 5.3.1 已确认：可以边问边要求证明材料

这一点已经确认。

v1.5 采用的不是：

`先把材料补齐，系统才开始真正问答`

而是：

`系统可以先进入实质问答，但会同步要求用户提供证明材料`

这代表产品要更像真人面签：

1. 面签官可以先听你解释
2. 也可以马上要求你提供证明
3. 如果你解释成立但没有证据，系统不能当成已经坐实
4. 如果你一直给不出证明，系统要把这件事当成风险点持续记住

所以 v1.5 的正确体验是：

`边问、边核、边要证据`

而不是：

`先走完补件流程，之后才开始真正思考`

## 5.3.2 已确认：会话过程中不主动做总结

这一点也已经确认。

v1.5 采用的不是：

`系统一边面谈，一边不断主动总结用户当前的问题点`

而是：

`系统在会话过程中不主动做总结，而是继续问、继续判断`

这代表：

1. 主体验仍然是面谈，不是报告讲解
2. 系统不需要频繁打断对话节奏
3. 用户不会一直被“当前状态总结”干扰

这也意味着：

1. 系统主要通过下一问来推进判断
2. 需要补证明时，系统直接说清楚要什么
3. 需要拒签时，再明确给出原因

简化理解就是：

`平时少总结，关键时刻再明确说明`

## 5.3.3 已确认：一次只追当前最关键的一个证明点

这一点也已经确认。

v1.5 采用的不是：

`系统一次性把所有缺的证明都抛给用户`

而是：

`系统一次只追当前最关键的一个证明点`

这代表：

1. 面谈节奏更像真人
2. 用户不会一下子被很多要求压住
3. 系统会先解决当前最影响判断的那个问题

这也意味着：

1. 系统要知道“现在最关键的缺口是什么”
2. 先把当前关键点问清楚、要清楚
3. 解决后再进入下一个问题

简化理解就是：

`一次只打一个关键点`

## 5.3.4 已确认：如果用户多轮都不给关键证明，系统可以进入不利结论

这一点也已经确认。

v1.5 采用的不是：

`只要用户还在聊天，系统就可以一直等下去`

而是：

`如果用户多轮都不给关键证明，系统可以进入不利结论`

这代表：

1. 系统不会无限等待
2. 关键证明长期缺失，本身就是风险
3. 用户反复解释但一直不给证明，不能被当成正常状态

这里的“不利结论”可以包括：

1. 当前风险明显升高
2. 当前判断对用户不利
3. 在严重情况下进入拒签

简化理解就是：

`关键证明长期拿不出来，本身就是结果的一部分`

## 5.3.5 已确认：如果用户一直回避核心问题，回避本身就是风险信号

这一点也已经确认。

v1.5 采用的不是：

`只要用户一直在说话，系统就当成是在正常配合`

而是：

`如果用户一直回避核心问题，回避本身就是风险信号`

这代表：

1. 系统不仅看用户有没有说话
2. 也看用户有没有真正回应关键问题
3. 如果用户一直绕开重点，系统不能假装这没有影响

这也意味着：

1. 回避核心问题会被记住
2. 回避次数增加，会让风险升高
3. 严重时可以进入不利结论

简化理解就是：

`一直不正面回答，本身就说明有问题`

## 5.4.6 已确认：高风险但未拒签，和已经拒签，必须分开

这一点也已经确认。

v1.5 采用的不是：

`只要风险很高，就直接等于拒签`

而是：

`高风险但未拒签，和已经拒签，是两个不同状态`

这代表：

1. 系统可以明确告诉用户“你现在风险很高”
2. 但只要还没有到拒签条件，就不能直接当成已经拒签
3. 拒签仍然是更重、更明确的最终结果

这样做的好处是：

1. 判断层次更清楚
2. 产品体验更接近真实面签
3. 系统不会因为“高风险”就过早收口

简化理解就是：

`高风险不等于已经拒签`

## 5.4 必须能拒签

系统不能把“拒签”做成一个非常难触发的边角状态。

如果出现明显红线，系统必须能进入拒签判断。

例如：

1. 明显造假
2. 关键事实前后冲突
3. 解释和材料无法自洽
4. 明显不符合该签证意图

当然，拒签不能随便发生。

但也不能变成“除非用户自己承认造假，否则基本不拒”。

## 5.4.1 已确认：明显红线可以直接拒签

这一点已经确认。

v1.5 采用的不是：

`所有高风险情况都必须再追问一次，才允许拒签`

而是：

`如果红线非常明显，系统可以直接拒签`

这意味着：

1. 系统不是无限兜底
2. 对明显造假、明显重大冲突、明显触线情况，不需要为了“像聊天”而强行再问一轮
3. 直接拒签本身也是面签体验的一部分

但这里要注意一个边界：

`只有红线非常明显时，才可以直接拒签`

如果只是可疑、解释不清、证据不足，那就不应该立刻拒签，而应该继续追问或要求补证明。

## 5.4.2 已确认：一旦拒签，当前会话立刻结束

这一点也已经确认。

v1.5 采用的不是：

`拒签后还在同一场会话里继续解释、继续挽回`

而是：

`一旦拒签，这场会话立刻结束`

这代表产品要更接近真实面签：

1. 拒签不是普通状态切换
2. 拒签意味着这次面谈已经结束
3. 用户如果还想重新尝试，应该进入新的会话

这样做的好处是：

1. 产品边界清楚
2. 不会让“拒签”变成一个不严肃的标签
3. 系统更容易让用户理解这次判断已经完成

## 5.4.3 已确认：新会话允许复用旧材料，但判断必须重新做

这一点也已经确认。

v1.5 采用的不是：

`只要开启新会话，用户就必须把所有材料重新上传一遍`

而是：

`新会话可以复用旧材料，但系统必须重新完成新的判断`

这代表：

1. 新会话在体验上不需要让用户重复劳动
2. 但新会话也不能简单继承上一次的结论
3. 旧材料可以继续作为证据输入
4. 新会话里的问答、追问、风险判断、拒签判断，都要重新发生

这样做的好处是：

1. 用户体验更好
2. 新会话仍然有独立意义
3. 系统可以避免“上次拒签结果直接粘贴到这次”的错误体验

这里补一条边界：

`如果以后保留“复用旧材料”能力，它也只能是显式导入的便利功能，不应变成跨会话的人物记忆。`

也就是说：

1. 新会话默认还是新的判断过程
2. 旧材料如果要进来，应该是明确选择
3. 旧结论、旧风险、旧拒签结果都不能自动带进来

## 5.4.4 已确认：用户可以随时主动上传材料，但系统必须明确反馈是否有帮助

这一点也已经确认。

v1.5 采用的不是：

`只有系统点名要求时，用户才能上传材料`

而是：

`用户可以随时主动上传材料，但系统必须明确告诉用户这份材料有没有帮助`

这代表：

1. 用户不需要被动等待系统开口
2. 用户可以主动补充自己认为重要的证明
3. 系统不能只是收文件，还要给出清楚反馈

这里的反馈至少要让用户知道：

1. 这份材料有没有帮助
2. 它支持了什么
3. 它没有解决什么问题
4. 如果不相关，也要明确说不相关

## 5.4.4.1 已确认：明显无关的材料，要直接说明没帮助，然后继续当前主线

这一点也已经确认。

v1.5 采用的不是：

`用户上传什么，系统都先含糊收下，不明确表态`

而是：

`如果材料明显无关，系统要直接告诉用户这份材料没帮助，然后继续当前主线`

这代表：

1. 系统不会被无关材料带偏
2. 用户能立刻知道这份材料有没有价值
3. 会话主线不会因为无关材料被打断

简化理解就是：

`不相关就直接说清楚，然后回到当前关键问题`

## 5.4.5 已确认：拒签后要告诉用户原因，但不暴露过深的内部细节

这一点也已经确认。

v1.5 采用的不是：

`拒签后只给一个冷冰冰的结果，不解释原因`

而是：

`拒签后要告诉用户原因，但不需要暴露太内部的判断细节`

这代表：

1. 用户需要知道为什么这次被拒
2. 系统不能只给结果不给说明
3. 但系统也不需要把所有内部规则、内部分数、内部链路全部摊开

用户层面至少应该能听懂：

1. 是因为材料冲突
2. 是因为关键证明缺失
3. 是因为发现明显红线
4. 是因为当前事实状态下风险过高

这样做的好处是：

1. 用户体验更清楚
2. 拒签更像一个完整判断，而不是黑盒结果
3. 也能为用户报告和复盘报告留出清晰边界

## 5.5 必须支持用提示词更新判断方式

很多签证判断不是硬编码能覆盖完的。

v1.5 必须支持：

1. 产品能改判断策略
2. 产品能改提问方式
3. 产品能改高风险识别方式
4. 这些调整尽量不靠改代码完成

## 5.5.1 已确认：签证官 Agent 的角色提示词必须方便产品修改

这一点要明确写死成需求。

当前项目里虽然已经有一些 `prompt_template_id` 之类的字段，
但它们更像标签，不是真正可由产品直接修改的提示词系统。

v1.5 采用的应该是：

`签证官 Agent 的角色提示词要变成可配置内容，而不是继续写死在代码里`

这代表：

1. 产品可以直接调整签证官的人设
2. 产品可以直接调整提问风格
3. 产品可以直接调整风险判断口径
4. 不需要每次都找开发改代码

产品真正需要能改的，不只是几个参数名，而是下面这些内容：

1. 签证官的角色定位
2. 面谈风格
3. 如何看待材料和口头解释
4. 哪些情况属于高风险
5. 哪些情况要继续追问
6. 哪些情况可以直接拒签

简化理解就是：

`角色提示词必须像内容配置一样可改，而不是像程序代码一样难改`

## 5.5.2 已确认：角色提示词后续要能参考真实案例，但 v1.5 先按规则写

这一点也很重要。

你前面已经说得很清楚：

1. 后续会有很多真实案例
2. 未来理论上要接 RAG
3. 但 v1.5 现在先不做案例检索

所以 v1.5 正确做法不是：

`现在就强行做案例库接入`

而是：

`先按规则把签证官角色提示词写对，同时为以后接真实案例留位置`

这代表：

1. v1.5 的角色提示词先根据规则、产品原则、常见风险模式来写
2. 提示词结构里要预留“真实案例参考”的位置
3. 以后接入案例时，不需要推翻整套提示词系统

简化理解就是：

`现在先按规则定义这个签证官，以后再让他参考真实案例成长`

## 5.5.3 v1.5 的提示词体系应该怎么设计

这里我不用技术语言，直接用产品能操作的方式来讲。

v1.5 不应该只有一大段混在一起的提示词。

更合理的做法是：

把签证官 Agent 的提示词拆成几块。

这样以后你要改的时候，不会一改全乱。

我建议至少拆成下面 5 块。

### 第一块：角色提示词

这一块回答：

`你到底是谁`

它主要定义：

1. 你是签证官，不是客服
2. 你要像真人面签一样判断
3. 你要持续记住对话和材料
4. 你不是来哄用户的
5. 你不是打分器，你是判断者

这一块是签证官的“人设底座”。

### 第二块：面谈风格提示词

这一块回答：

`你要怎么问`

它主要定义：

1. 语气是正式还是温和
2. 问题是一句一句追，还是一次说很多
3. 要不要直接引用材料
4. 遇到冲突时要不要直接点出来
5. 遇到不相关材料时怎么回应

这一块决定用户感受到的“像不像真人面签官”。

### 第三块：判断规则提示词

这一块回答：

`你看重什么`

它主要定义：

1. 什么算关键证明
2. 什么算高风险
3. 什么算明显红线
4. 什么情况继续问
5. 什么情况进入高风险
6. 什么情况可以直接拒签

这一块虽然是提示词，但内容现在先按规则来写。

也就是说：

`先把产品规则写成签证官的判断口径`

### 第四块：输出方式提示词

这一块回答：

`你最后怎么说`

它主要定义：

1. 平时不要主动做总结
2. 一次只追一个最关键点
3. 要求补证明时要说清楚为什么
4. 拒签时要说明原因，但不要暴露太多内部细节

这一块是为了保证用户真正看到的体验稳定。

### 第五块：未来案例参考位

这一块现在先不真正接案例库。

但提示词结构里要预留一个位置，未来可以放：

1. 真实案例摘要
2. 常见拒签模式
3. 常见高风险情形
4. 某类签证下的典型判断经验

这样以后接真实案例时，不需要把前 4 块全部推翻。

## 5.5.4 产品以后真正会改的，不是一个词，而是一套“签证官说明书”

你以后真正会改的，不应该只是：

1. 一个模型名
2. 一个 prompt id
3. 一个版本号

你真正需要改的是一套“签证官说明书”。

我建议你以后把它理解成下面 6 个可改模块：

1. 这个签证官是谁
2. 这个签证官怎么问
3. 这个签证官怎么判断风险
4. 这个签证官什么时候继续问
5. 这个签证官什么时候要求证明
6. 这个签证官什么时候直接拒签

如果未来能把这 6 个模块分开改，你作为产品就会舒服很多。

## 5.5.5 v1.5 先按规则写提示词，但写法要像真人面签官

这一点很重要。

现在虽然还不接真实案例，
但也不能把提示词写成纯 checklist。

v1.5 更合理的写法应该是：

1. 背后依据是规则
2. 对外表现像真人面签官
3. 问答方式是连续的
4. 判断方式是围绕冲突、证明、风险推进的

简单说：

`底层先按规则，表层要像面签官`

这样以后接真实案例时，才是顺着长出来，不是重新换脑子。

## 5.6 打分只能做辅助，不能做主角

打分可以保留，但只能是辅助信号。

打分的作用应该是：

1. 帮助系统判断风险高低
2. 帮助系统发现还缺什么
3. 帮助系统整理内部状态

打分不应该变成产品主体验。

用户不应该感受到自己在和一个打分器交流。

## 6. v1.5 暂时不做什么

为了把方向拉正，v1.5 也要明确不做什么。

## 6.1 暂时不做外部案例 RAG

你已经说得很清楚：

真实案例后面很重要，但 v1.5 先不做。

所以 v1.5 只要求：

1. 给以后接案例库预留位置
2. 不要求现在就把案例检索做完

## 6.2 暂时不做很重的平台化能力

v1.5 不追求：

1. 很复杂的工作台
2. 很全的模型平台
3. 很大的通用框架

当前目标是先把“Agent 面签官”这件事做对。

## 6.3 暂时不追求所有签证都做深

v1.5 可以先把核心签证家族做好。

关键不是覆盖最多，而是先把一条完整体验做真。

## 6.4 v1.5 的 MVP 范围

这一节非常重要。

因为 v1.5 如果不切范围，很容易又回到“什么都想做，最后主线做歪”的状态。

所以这里直接定义：

`MVP 的目标不是做最全，而是先把“像真人面签官”这件事做真。`

## 6.5 v1.5 这版必须做的

下面这些是 MVP 必做项。

少一项，产品体验都会明显跑偏。

### 6.5.1 连续会话必须成立

必须做到：

1. 同一场会话里持续记住用户说过的话
2. 持续记住用户上传过的材料
3. 后面的提问必须建立在前面内容上

这是最基础的一条。

没有它，就不是 Agent 面签官。

### 6.5.2 会话中必须能边问边核，边要证明

必须做到：

1. 材料没补齐也能进入实质问答
2. 系统能同步要求关键证明
3. 一次只追当前最关键的一个点

这条是整个产品节奏的核心。

### 6.5.3 系统必须能把“话”和“材料”放在一起判断

必须做到：

1. 系统能记住用户说法
2. 系统能记住材料内容
3. 系统能发现两者冲突
4. 系统追问时能明确引用材料

没有这条，就还是“聊天系统”和“材料系统”各干各的。

### 6.5.4 系统必须有清楚的风险分层

必须做到：

1. 正常问答
2. 正在核实关键问题
3. 等待关键证明
4. 高风险但未拒签
5. 已拒签并结束

这样产品才不会把所有情况混成一团。

### 6.5.5 系统必须能拒签，而且拒签逻辑要像真人面签

必须做到：

1. 明显红线可以直接拒签
2. 拒签后当前会话立刻结束
3. 拒签后要告诉用户原因
4. 但不暴露太深的内部细节

### 6.5.6 用户可以主动上传材料，系统必须明确反馈

必须做到：

1. 用户可以随时主动上传
2. 系统能告诉用户这份材料有没有帮助
3. 如果无关，要直接说明无关
4. 不让主线被无关材料带偏

### 6.5.7 签证官角色提示词必须可配置

必须做到：

1. 产品能修改签证官角色提示词
2. 产品能修改提问风格
3. 产品能修改判断口径
4. 不需要每次都改代码

这是 v1.5 和当前版本最关键的差别之一。

### 6.5.8 打分必须退到辅助位置

必须做到：

1. 打分只做内部辅助
2. 不再做用户主体验
3. 不再替代 Agent 的主判断

## 6.6 v1.5 这版可以延后的

下面这些重要，但可以放到后面。

它们不是现在的第一优先级。

### 6.6.1 真实案例 RAG

重要，但这版先不做。

这版只需要：

1. 给以后留接口
2. 给提示词结构预留案例参考位

### 6.6.2 更复杂的多签证深度策略

这版不用一开始就把所有签证家族都做很深。

可以先把：

1. 核心签证家族
2. 核心高风险场景
3. 核心补证明链路

跑通。

### 6.6.3 很复杂的前端工作台

这版不需要做成很重的系统平台。

重点是：

1. 面谈体验对
2. 判断逻辑对
3. 提示词可改

### 6.6.4 很细的内部分析报表

这版不需要优先做很复杂的分析大盘。

先把：

1. 用户能正常走完整流程
2. 拒签和高风险有清楚边界
3. 系统能引用材料追问

做好更重要。

## 6.7 v1.5 明确不应该现在做的

下面这些事情，现在做了反而容易让方向继续跑偏。

### 6.7.1 不要再把 OpenAI 兼容当主目标

兼容入口重要，但它只是入口。

主目标永远是：

`Agent 面签官体验成立`

### 6.7.2 不要再把打分系统当主骨架

打分可以留，
但不能再继续围着打分去设计整个产品。

### 6.7.3 不要为了“看起来很完整”而堆太多功能

如果主线还不对，
功能越多，越容易把问题藏起来。

### 6.7.4 不要做跨会话的人物长期记忆

当前产品不需要把用户做成长期跟踪对象。

所以这版不要做：

1. 用户历史画像长期积累
2. 多次会话自动串联
3. 基于旧会话直接影响新会话的隐藏判断

当前这版只需要把：

`本次会话的连续记忆`

做好。

## 6.8 我建议的 MVP 交付标准

如果要一句话定义 v1.5 是否交付成功，
我建议用下面这个标准：

`用户能明显感觉自己在和同一个会记忆、会看材料、会追问、会收口的面签官对话。`

更具体一点，可以拆成 5 条：

1. 用户前面说过的话，后面不会被忘
2. 用户上传的材料，后面会被引用来追问
3. 系统一次只追一个当前最关键的问题
4. 高风险和拒签是两个不同状态
5. 红线明显时，系统真的能直接拒签并结束会话

## 7. v1.5 的理想用户流程

下面是我们希望用户经历的主流程。

### 第一步：进入会话

用户进入系统，系统知道这是同一场面签的开始。

### 第二步：上传初始材料

用户上传已有材料。

系统不只是收文件，而是开始理解这些材料表达了什么。

用户在后续问答过程中，也可以随时主动上传新的材料。

系统不能只是收下文件。

系统必须明确反馈：

1. 这份材料有没有帮助
2. 它支持了什么判断
3. 它还没有解决什么问题
4. 如果不相关，要直接说明

### 第三步：开始问答

系统根据：

1. 用户已经说过的话
2. 用户已经上传的材料
3. 当前签证类型

来决定下一问。

即使材料还不完整，也可以开始问。

但系统必须同步告诉用户：

1. 当前缺什么证明
2. 为什么这个证明重要
3. 如果一直没有证明，会带来什么风险

### 第四步：动态追问与补充

如果材料不够，系统要求补充。

如果解释不清，系统继续追问。

如果材料和解释冲突，系统重点追问冲突点。

### 第五步：形成判断

系统根据整场会话形成当前判断：

1. 可继续
2. 需要补强
3. 高风险
4. 拒签

## 7.1 v1.5 的完整用户流程

上面是简版主流程。

如果用更完整的产品方式来讲，v1.5 的体验应该是下面这样。

### 阶段 1：进入会话

用户进入系统。

系统完成两件事：

1. 建立这次面签会话
2. 记住这是一场新的、独立的判断过程

这时系统不会假装已经了解用户。

它只是准备开始听、开始看、开始判断。

### 阶段 2：用户先说，也可以先传

用户可以：

1. 先介绍自己的情况
2. 先上传已有材料
3. 两者混着来

这里不要求必须先把所有材料补齐。

系统可以一边听，一边看，一边进入实质问答。

### 阶段 3：系统抓当前最关键问题

系统不会一次抛很多要求。

系统要先判断：

`当前最影响结果的那个问题是什么`

比如：

1. 资金来源说不清
2. 出行目的解释不清
3. 材料和口供冲突
4. 关键证明一直没给

然后系统只围绕这个最关键问题继续追问。

### 阶段 4：系统边问边核

这是 v1.5 最重要的体验。

系统会同时做 3 件事：

1. 听用户现在怎么说
2. 对照前面用户说过什么
3. 对照已经上传的材料写了什么

如果发现：

1. 解释和材料一致
2. 解释虽然新，但逻辑说得通
3. 用户补上了关键证明

那系统可以继续推进。

如果发现：

1. 解释和材料冲突
2. 用户一直回避问题
3. 用户反复改口但讲不通
4. 关键证明一直不给

那系统会把风险逐步拉高。

### 阶段 5：形成阶段性结论

系统在每个阶段，其实都在做判断。

但对用户来说，结论只需要体现成这几种：

1. 继续问
2. 当前要补一个关键证明
3. 当前已经进入高风险
4. 当前直接拒签

这里最重要的是：

`高风险不等于已经拒签`

只有红线非常明显时，才直接拒签。

### 阶段 6：结束或重新开始

如果系统进入拒签：

1. 当前会话立刻结束
2. 系统要告诉用户原因
3. 但不展开过深的内部细节

如果用户要重新开始：

1. 可以开新会话
2. 可以复用旧材料
3. 但新的判断必须重新做

## 7.2 v1.5 的状态定义

为了让产品清楚，我们需要把系统状态说得非常简单。

我建议 v1.5 至少有下面 5 个用户可理解状态。

### 状态 1：正常问答中

意思是：

1. 系统还在继续面谈
2. 还没有进入明显风险收口
3. 当前只是继续核实和推进

用户感受到的是：

`系统还在正常问我`

### 状态 2：正在核实关键问题

意思是：

1. 系统已经发现一个关键点
2. 但还没有到拒签
3. 现在要围绕这个点继续问、继续要证明

比如：

1. 资金来源不清楚
2. 材料与解释有冲突
3. 某个重要事实还没被证明

用户感受到的是：

`系统现在盯住了一个关键问题`

### 状态 3：等待关键证明

意思是：

1. 系统已经明确知道现在最缺什么
2. 没有这个证明，判断就很难往前走
3. 系统仍然可以继续问，但风险会跟着累积

用户感受到的是：

`系统现在最想看的是某一个关键证明`

### 状态 4：高风险但未拒签

意思是：

1. 风险已经很高
2. 系统已经明显不放心
3. 但还没有到最终拒签条件

这时系统仍然可以：

1. 再追问一次
2. 再要一次关键证明
3. 再核实一次重大冲突

用户感受到的是：

`系统已经很怀疑我了，但还没有正式结束`

### 状态 5：已拒签，已结束

意思是：

1. 这次面签判断已经完成
2. 当前会话不能继续
3. 系统会告诉用户拒签原因

用户感受到的是：

`这次已经结束了`

## 7.3 v1.5 的判断分层

这部分是为了避免系统再做歪。

系统的判断不要混成一团。

我建议分成 3 层。

### 第一层：事实层

这一层只回答：

`现在我们到底知道了什么`

包括：

1. 用户说了什么
2. 用户前面说过什么
3. 用户上传了什么材料
4. 材料里明确写了什么

这一层不急着下结论。

先把事实记住。

### 第二层：核实层

这一层只回答：

`这些事实之间有没有问题`

主要看：

1. 有没有冲突
2. 有没有回避
3. 有没有关键证明缺失
4. 新解释能不能讲通
5. 材料能不能支持当前说法

这一层的作用是：

把问题点找出来。

### 第三层：结论层

这一层才回答：

`现在该怎么处理`

只做 4 类结论：

1. 继续问
2. 要一个关键证明
3. 进入高风险
4. 直接拒签

这层不要太复杂。

对用户来说，最终只要能感知到这 4 类处理结果就够了。

## 7.4 打分在 v1.5 里的正确位置

这一点需要特别强调。

打分不是主线。

打分只应该放在内部辅助层。

它的作用是：

1. 帮系统判断当前风险高不高
2. 帮系统判断现在最该追哪一个证明点
3. 帮系统整理内部优先级

它不应该直接决定：

1. 该问什么的全部逻辑
2. 是否已经拒签
3. 对用户说什么

简化理解就是：

`打分是后台参考，不是前台主角`

## 7.5 一个简单例子

为了让这个流程更好理解，下面举一个简单例子。

### 例子

用户说：

`我去美国读书，父母出钱。`

系统看到：

1. 用户说父母出钱
2. 目前还没有资金证明
3. 材料里暂时也没有能证明这件事的内容

这时正确流程不是：

1. 直接相信
2. 直接拒签
3. 一次性抛出很多要求

而是：

1. 先继续问
2. 只盯当前最关键问题，也就是资金来源证明
3. 明确要求用户提供相关证明
4. 如果用户 later 补上证明，继续往前走
5. 如果用户多轮解释但一直不给证明，风险升高
6. 如果同时又出现前后说法冲突，进一步进入高风险

这个例子就是 v1.5 想要的感觉：

`像一个会记忆、会核实、会追证据的面签官`

## 8.1 用户层验收

2. 用户会明确感觉系统记得前面上传的材料

## 8.3 系统层验收

1. 同一场会话有稳定上下文
2. 材料和解释能进入同一条判断链
3. 系统保留证据来源，知道为什么这样判断
4. 后续要接真实案例库时，不需要推翻重来

## 9. v1.5 的核心原则

为了避免再做歪，v1.5 需要守住 5 条原则。

### 原则 1：对话优先

主体验是连续面谈，不是流程跳转。

### 原则 2：证据优先

判断必须尽量有材料或上下文支持。

### 原则 3：解释也算事实输入

用户的话不能一轮就丢。

### 原则 4：规则做护栏，不做主角

规则负责兜底，不应该取代面签官本身。

### 原则 5：策略可迭代

提示词、判断口径、追问方式，都要方便后续调整。

## 13. v1.5 系统结构图（产品版）

这一节不用技术图纸的方式讲，
只用产品能理解的方式讲清楚：

`这套系统到底分成哪几部分，它们分别干什么。`

## 13.1 一句话总结构

v1.5 的正确结构应该是：

`本次会话记忆层 -> 材料理解层 -> 面签官判断层 -> 用户结果层`

也就是说：

1. 先把本次会话里发生的内容记住
2. 再把材料转成系统能理解的证据
3. 再交给面签官 Agent 做追问和判断
4. 最后把结果用用户能理解的话说出来

## 13.2 第一层：本次会话记忆层

这一层只负责一件事：

`记住这次会话里发生了什么`

它记住的不是“这个用户是谁”，而是：

1. 这次会话里用户说过什么
2. 这次会话里系统问过什么
3. 这次会话里用户上传过什么材料
4. 这次会话里已经出现了哪些冲突
5. 这次会话当前最关键的问题是什么

这一层的作用是：

1. 防止系统下一轮忘记前面内容
2. 让后面的提问建立在前面的基础上
3. 让整场面谈像同一个签证官在持续追问

这里最重要的一条边界是：

`这层只记住本次会话，不做跨会话的人物记忆`

## 13.3 第二层：材料理解层

这一层只负责一件事：

`把用户上传的材料变成系统能拿来判断的证据`

用户上传的是文件。

但面签官真正需要的不是“一个文件名”，而是：

1. 材料里写了什么
2. 哪些内容支持用户说法
3. 哪些内容和用户说法冲突
4. 哪些关键点其实还没被证明

所以这一层的作用是：

1. 读取材料
2. 抽出关键内容
3. 形成证据点
4. 告诉系统这份材料有没有帮助

简单说：

`它负责把文件变成证据`

## 13.4 第三层：面签官判断层

这一层是产品真正的核心。

它负责：

1. 看本次会话里用户说过什么
2. 看当前已经有哪些材料和证据
3. 比较两边有没有冲突
4. 决定下一步最该追哪一个问题
5. 决定当前是继续问、要证明、进入高风险、还是拒签

这一层就是“签证官 Agent”本体。

它不是客服。

它不是打分器。

它也不是材料清单机器人。

它更像一个真人签证官在做这几件事：

1. 听你说
2. 看你交的材料
3. 比较前后逻辑
4. 抓住当前最关键问题
5. 作出阶段性判断

## 13.5 第四层：用户结果层

这一层只负责一件事：

`把系统判断变成用户能理解的表达`

用户不应该直接看到很多内部结构。

用户真正需要看到的是：

1. 下一问是什么
2. 现在要补什么证明
3. 当前是不是高风险
4. 如果拒签，原因是什么

所以这一层要做的是：

1. 平时不主动做很多总结
2. 继续用自然的面签问法推进
3. 关键时刻说明原因
4. 拒签时清楚说明，但不暴露太深内部细节

## 13.6 数据库在这套结构里的正确位置

这一点要特别说清楚。

数据库在 v1.5 里的正确位置，不是：

`替代面签官做判断`

而是：

`作为本次会话记忆和材料证据的存放容器`

也就是说：

1. 数据库负责把本次会话内容存下来
2. 数据库负责把材料和证据存下来
3. 数据库负责让系统下一轮还能拿到这些内容

但数据库本身不应该变成：

`一个用户看不见、却能直接决定拒签的黑盒来源`

所以正确关系是：

1. 数据库存放本次会话事实
2. 面签官 Agent 从这些事实里取当前需要的内容
3. 再基于本次会话里的内容做判断

简化理解就是：

`数据库是本次会话的案卷夹，不是隐藏裁判`

## 13.7 为什么不能只靠模型上下文

这里也用产品语言说。

如果只靠模型当前这一轮上下文，会有 4 个问题：

1. 聊多了以后容易忘
2. 材料多了以后不方便一直全塞进去
3. 上传文件、解析材料、形成证据这些过程需要有地方存
4. 最后要说明拒签原因时，也需要能回到这次会话里的依据

所以 v1.5 更合理的方式不是：

`只有上下文，没有存放`

而是：

`先把本次会话内容存好，再把当前最相关的内容拿给面签官 Agent 用`

## 13.8 为什么现在不需要向量库做主底座

对 v1.5 来说，当前最重要的不是“找相似案例”。

当前最重要的是：

1. 把本次会话记住
2. 把本次材料记住
3. 把本次判断主线走对

所以这版里，主底座应该还是：

`本次会话记忆 + 本次材料证据`

而不是一上来就把重点放到案例检索上。

以后如果要接真实案例，
向量库更适合作为：

`补充检索层`

而不是直接替代这次会话的主底座。

## 13.9 这套结构最重要的产品边界

如果把这套结构压缩成最重要的 3 句话，就是：

1. `系统只记住本次会话，不做跨会话人物记忆`
2. `拒签依据必须来自本次会话里的资料和对话`
3. `数据库是本次会话的存放容器，签证官 Agent 才是判断者`

只要这 3 条不偏，v1.5 就不会再轻易做歪。
